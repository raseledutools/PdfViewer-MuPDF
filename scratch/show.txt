    Fix 2 real bugs: Firebase keyword sync delay + floating windows missing keyword block
    
    Bug 1 — Firebase keyword update doesn't apply until app restart:
    FirebaseKeywordSync.fetchFromFirebase() used addListenerForSingleValueEvent
    — fetches ONCE, never updates again on its own. The code comment even
    admitted this ('data change হলে forceRefresh call করো') but nothing
    anywhere actually calls forceRefresh() automatically. So updating a
    keyword in the Firebase console had zero effect until the app process was
    killed and restarted.
    
    Fix: switched to addValueEventListener — Realtime Database's actual live
    listener, fires automatically on every change at that path. The earlier
    switch away from it was guarding against a listener leak, which is a real
    concern for a listener tied to a short-lived Activity/Fragment, but this
    is a permanent app-process-lifetime singleton with nothing to leak past.
    Also fixed activeListener never actually being assigned (dead code before
    this), so forceRefresh()/repeated calls now properly detach the old
    listener before attaching a new one instead of accumulating duplicates.
    
    Bug 2 — floating YouTube / Facebook / generic floating window: adult
    keyword typed into a search box doesn't trigger any block screen, even
    though the full-page browser catches the exact same keyword:
    YoutubeFloatingWindowService.kt, FacebookFloatingWindowService.kt, and
    FloatingWindowService.kt's WebViewClients only ever called
    AdBlocker.isAdultSite(url) — a DOMAIN-only check. Typing an adult keyword
    into a search box on a legitimate domain (youtube.com, facebook.com)
    never matches a domain check, unlike the full-page browser which also
    checks the URL text itself against FirebaseKeywordSync's keyword list
    (catches it because the search query ends up embedded in the URL, e.g.
    .../results?search_query=X).
    
    Fix: added FirebaseKeywordSync.containsAdultKeyword(url) alongside the
    existing isAdultSite(url) check in shouldInterceptRequest AND
    shouldOverrideUrlLoading for all three files, and upgraded the blank/silent
    response to a real blocked page (AdBlocker.buildBlockedPage() for the two
    service files; FloatingWindowService.kt already had its own
    buildFloatingBlockPage(), reused as-is).
    
    On the reported slowness: I don't have profiling data or a crash log
    pointing at a specific cause, so I'm not guessing at a fix without
    evidence. One plausible contributor: bug 1 meant restarting the app was
    the ONLY way to pick up new keywords — if that's been happening
    repeatedly, repeated cold starts would genuinely feel slower than a warm,
    properly-running app. Worth re-checking after this fix; if it's still
    slow, a fresh crash/ANR log would let me look at an actual cause instead
    of guessing.
